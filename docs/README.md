# Canoniq docs

This directory is written to be **dropped straight into a Docusaurus
`docs/` folder** with no rework: every page has Docusaurus frontmatter
(`id`, `title`, `sidebar_position`, `description`), and each subfolder has
a `_category_.json` controlling its sidebar label and position.

Until then, it's also perfectly readable as plain markdown on GitHub or in
an editor — start at [`intro.md`](./intro.md).

```
docs/
  intro.md                              — what canoniq is, positioning, non-goals summary
  getting-started.md                    — install + quickstart, both pipelines

```

## Standing up the actual Docusaurus site (when you're ready)

This repo doesn't include a Docusaurus install yet — that's a separate
step, deliberately not done until you want a publishable site. When you
do:

```bash
# from the repo root, or a separate docs-site/ checkout
npx create-docusaurus@latest canoniq-site classic --typescript
```

Then either:

1. **Point Docusaurus at this folder directly** — in
   `canoniq-site/docusaurus.config.js`, set the `docs` plugin's `path` to
   the absolute/relative path of this `my-semantic-layer/docs/` directory
   (or symlink it into `canoniq-site/docs/`). This keeps documentation
   changes living alongside the code they describe, which is the better
   long-term setup for a project that changes as fast as this one has.
2. **Or copy the contents into `canoniq-site/docs/`** if you want the
   docs site to be a fully separate repo/deploy target.

Either way, exclude this `README.md` from the docs glob (Docusaurus's
default `docs` plugin option `exclude: ['**/README.md']` or similar), since
it has no Docusaurus frontmatter and isn't meant to be a rendered page —
it's contributor-facing only.

A minimal `docusaurus.config.js` sidebar entry, once wired up:

```js
// sidebars.js
module.exports = {
  docsSidebar: [{type: 'autogenerated', dirName: '.'}],
};
```

The `_category_.json` files already set each section's position (`intro`
and `getting-started` are top-level pages at positions 1–2; `architecture`
is 3, `guides` is 4, `emitters` is 5, `reference` is 6), so the
autogenerated sidebar will come out in the right order with no further
configuration.

## Keeping these docs current

These docs describe the codebase as of the CANONIQ_V1_REPORTFIRST_SPEC
implementation. If you change:

- the fingerprint expression grammar or confidence-band thresholds → update
  [`architecture/numeric-fingerprinting.md`](./architecture/numeric-fingerprinting.md)
- the `SourceType` trust table → update
  [`architecture/ontorank-trust.md`](./architecture/ontorank-trust.md)
- a CLI flag or command → update
  [`guides/cli-reference.md`](./guides/cli-reference.md)
- the benchmark's planted traps or scorecard fields → update
  [`guides/benchmark.md`](./guides/benchmark.md)

The root [`README.md`](../README.md) stays the short/marketing-facing
overview; these docs are the deeper reference.
